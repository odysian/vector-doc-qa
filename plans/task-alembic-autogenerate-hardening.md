## Summary

Harden Alembic autogenerate/check behavior for `quaero` schema so schema reflection is deterministic across local/prod-like environments and does not propose noisy or dangerous migration ops.

## Problem Statement

`alembic check` currently reports false-positive operations such as:

1. remove/add of FKs where only schema qualification differs (`users.id` vs `quaero.users.id`)
2. attempts to remove `alembic_version`
3. noisy add/drop churn when schema reflection resolves tables inconsistently (`schema=None` vs `schema=quaero`)

This has happened repeatedly and increases the risk of generating poor migration files.

## Goals

1. Make autogenerate/check deterministic for Quaero-managed objects.
2. Prevent unsafe autogenerated migrations by default.
3. Keep behavior safe for both local and production migration workflows.

## Non-Goals

1. Re-architecting database schema layout.
2. Splitting into multiple Alembic environments unless hardening fails.
3. Editing historical applied migration files.

## Deliverables

1. `backend/alembic/env.py` hardening with a canonical schema-reflection contract:
   - explicit handling of `include_schemas`, `version_table_schema`, and schema filters
   - filtering logic that keeps only `quaero` app objects and avoids default-schema false positives
   - clear inline comments explaining why each guard exists
2. `process_revision_directives` safety guardrails in `env.py`:
   - block or fail fast on unexpected `drop_table` autogen ops
   - block ops outside `quaero` schema
   - block suspicious FK churn where only schema qualification changed
3. Documentation update in `docs/PATTERNS.md` (and `docs/ARCHITECTURE.md` if needed):
   - migration autogenerate rules and review policy
   - do-not-merge criteria for autogenerated migrations
4. Verification recipe committed to docs/task notes for repeatable validation.

## Acceptance Criteria

- [ ] `alembic check` at head (no model changes) exits clean with no false-positive upgrade operations.
- [ ] `alembic revision --autogenerate` at head yields no meaningful ops for unchanged models.
- [ ] No autogenerated operation targets non-`quaero` schemas.
- [ ] No autogenerated operation proposes dropping `alembic_version`.
- [ ] Guardrails fail fast when intentionally forcing a dangerous autogen diff.
- [ ] Existing migration flow remains valid:
  - `alembic upgrade head`
  - `alembic downgrade -1`
  - `alembic upgrade head`
- [ ] `make backend-verify` passes.

## Implementation Plan

1. Reproduce and snapshot current `alembic check` false positives for baseline.
2. Implement schema/reflection hardening in `backend/alembic/env.py` using a minimal, explicit rule set.
3. Add `process_revision_directives` guardrails to reject dangerous/noisy autogenerated directives.
4. Run migration verification (`upgrade/downgrade/upgrade`) and `alembic check`.
5. Update docs with the migration policy and troubleshooting notes.

## Verification

```bash
cd backend

# Baseline migration health
alembic upgrade head
alembic downgrade -1
alembic upgrade head

# Drift/autogen guard checks
alembic check
alembic revision --autogenerate -m "tmp-autogen-smoke"
# verify generated script is empty/safe, then delete temp revision

# Full backend checks
cd ..
make backend-verify
```

## Risks and Mitigations

1. Over-filtering could hide legitimate diffs.
   - Mitigation: keep filters minimal and schema-targeted, validate with intentional test changes.
2. Guardrails could block valid migrations.
   - Mitigation: allow explicit override path and document review process.
3. Environment-specific search_path differences could still leak in.
   - Mitigation: enforce canonical migration invocation and document required DB/session settings.
